# Architecture Design

## System Layers

1. Algorithm Service (`algorithm/`)
- Computes combat damage and multipliers
- Generates AI decisions
- Provides validation and skill-tree related calculations

2. Strategy Service (`strategy/`)
- Stores and enforces game rules
- Maintains world state and game events
- Handles player/world query and update flows

3. Electron Frontend (`electron-client/`)
- Renders UI (main menu, gameplay, inventory, settings)
- Captures input and updates local UI state
- Uses Electron IPC to call gRPC methods via the main process

## Data Flow

1. Player interacts in Electron renderer
2. Renderer invokes preload API
3. Preload forwards request to Electron main process via IPC
4. Main process calls backend gRPC service
5. Response is returned to renderer and reflected in UI

## gRPC Contracts

- `AlgorithmService.proto`
  - `CalculateDamage`
  - `GetSkillTree`
  - `AIActionDecision`
  - `ValidateInput`

- `StrategyService.proto`
  - `GetGameRules`
  - `UpdateWorldState`
  - `TriggerEvent`
  - `QueryGameState`

## Deployment Model

- Run two backend services as native executables
- Run Electron as desktop client
- Optional packaging for Electron can be added later

## Legacy Note

`application/` contains the previous Qt implementation.
It is retained only for migration fallback and should not be treated as the primary frontend.
